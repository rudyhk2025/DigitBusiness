# 数字员工 · 开发执行计划

基于 [product.md](./product.md) 的路线图，拆解为可执行任务与交付物。

---

## 0. 前置准备（Day 0）

| 序号 | 任务 | 产出 | 备注 |
|------|------|------|------|
| 0.1 | 创建 Python 3.10+ 虚拟环境 | `venv` / `requirements.txt` | 锁定 Playwright、openai/国产 API 客户端版本 |
| 0.2 | 安装 Playwright 并下载 Chromium | 可启动无头/有头浏览器 | 需 `playwright install chromium` |
| 0.3 | 集成 playwright-stealth | 可用的 stealth 启动方式 | 反爬伪装基础 |
| 0.4 | 设计项目目录结构 | 见下方「建议目录」 | 模块化、易扩展 |

**建议目录结构：**

```
DigitBusiness/
├── src/
│   ├── env/           # 模块1：多平台环境管理器
│   ├── filter/        # 模块2：精准筛选引擎（抖音/京东/小红书）
│   ├── chat/          # 模块3：AI 对话与邀约
│   ├── report/        # 模块4：信息上报与通知
│   ├── db/            # SQLite 模型与迁移
│   └── config/        # 平台配置、频率限制、API Key 等
├── data/              # user_data_dir 等运行时数据（不入库）
├── doc/
├── tests/
├── requirements.txt
└── README.md
```

## 1. 第一阶段：自动化地基（第 1–3 天）

### 1.1 多平台环境管理器（对应 product 模块 1）

| 序号 | 任务 | 产出 | 验收 |
|------|------|------|------|
| 1.1.1 | 实现 `BrowserManager` 类 | `src/env/browser_manager.py` | 可指定平台（DY/JD/XHS）启动 Chromium |
| 1.1.2 | 为各平台配置独立 `user_data_dir` | 配置表或常量 | 三平台路径隔离，Cookies 不互窜 |
| 1.1.3 | 集成 stealth，抹除 webdriver 特征 | 启动时应用 stealth | 本地验证无 `navigator.webdriver` |
| 1.1.4 | 实现「免登录启动」流程说明/脚本 | 文档 + 可选登录辅助脚本 | 首次需人工登录一次，之后可复用 Session |

### 1.2 精准筛选引擎（对应 product 模块 2）

| 序号 | 任务 | 产出 | 验收 |
|------|------|------|------|
| 1.2.1 | 抖店：进入精选联盟 + 筛选（带货分/粉丝数/类目） | `src/filter/douyin.py` | 能输出一页达人列表（昵称、ID、关键指标） |
| 1.2.2 | 京麦：进入达人平台 + 筛选（转化率/客单价） | `src/filter/jd.py` | 能输出一页达人列表 |
| 1.2.3 | 蒲公英：博主广场 + 最近 3 篇笔记标题 | `src/filter/xiaohongshu.py` | 能输出博主 + 3 条笔记标题，供后续 AI 用 |
| 1.2.4 | 微信小店助手：达人列表页监听接口 + 解析基础信息 | `src/filter/wechat_channels.py` | 能输出一页达人列表（uid、昵称、粉丝数等） |
| 1.2.5 | 统一「达人搜索结果页」数据模型 | 如 `TalentCandidate` dataclass | 多平台解析后映射到统一结构，便于入库与对话 |

### 1.3 数据库与存储

| 序号 | 任务 | 产出 | 验收 |
|------|------|------|------|
| 1.3.1 | 按平台分表建 SQLite | `src/db/schema.sql` | talent_dy, talent_jd, talent_xhs 各平台独立字段 |
| 1.3.2 | 实现各平台 repo | `src/db/talent_dy_repo.py` 等 | add, get_by_uid, update_status, update_chat_log, update_contact |

**第一阶段结束标志：** 可一键选择平台 → 启动浏览器 → 抓取一页达人 → 写入/更新 SQLite，且三平台互不干扰。

## 2. 第二阶段：AI 逻辑集成（第 4–6 天）

### 2.1 API 与 Prompt（对应 product 模块 3 前半）

| 序号 | 任务 | 产出 | 验收 |
|------|------|------|------|
| 2.1.1 | 接入 OpenAI 或国产大模型 API | `src/chat/llm_client.py` | 统一接口：输入 prompt + 可选 system，返回回复文本 |
| 2.1.2 | 设计 Prompt 模板库（按平台分层） | `src/chat/prompts/` 或配置 | 小红书偏亲切、京东偏专业；含「初次邀约」多种风格 |
| 2.1.3 | 实现「初次邀约」：3–5 种风格钩子文案，AI 随机选一条发送 | 邀约生成函数 + 发送逻辑 | 文案不重复、可配置随机种子 |

### 2.2 对话与意图识别

| 序号 | 任务 | 产出 | 验收 |
|------|------|------|------|
| 2.2.1 | 解析达人回复（要样机/问佣金/拒绝/加微信等） | 意图分类或关键词+AI 结合 | 可输出结构化意图 + 置信度/原文 |
| 2.2.2 | 深度谈判：根据意图调用不同策略（如佣金加码、独家机制） | 策略路由 + 回复生成 | 与「最高权限」等配置联动，可配置化 |
| 2.2.3 | 自动回复轮询：每 15 分钟扫描未读消息 | 定时任务或循环 + 调用各平台私信接口/页面 | 新消息 → 意图识别 → 生成回复 → 发送 → 更新 chat_log |

### 2.3 闭环与上报（对应 product 模块 4）

| 序号 | 任务 | 产出 | 验收 |
|------|------|------|------|
| 2.3.1 | 成功判定：识别「微信号」或「明确合作意向」 | 规则 + 可选 AI 二次确认 | 触发时写 contact、更新 status |
| 2.3.2 | 格式化上报内容（主页链接、粉丝数、联系方式） | `src/report/formatter.py` | 固定模板，便于邮件/Webhook 使用 |
| 2.3.3 | 邮件推送：status 变更为「成交」时 `send_email()` | `src/report/notifier.py`（smtplib） | 可配置收件人、发件人、正文模板 |
| 2.3.4 | 可选：飞书/钉钉 Webhook | 同一 notifier 接口，多通道 | 配置开关选择邮件或 Webhook |

**第二阶段结束标志：** 从邀约 → 多轮对话 → 识别意向/微信号 → 更新 DB → 触发邮件/Webhook 全链路跑通。

## 3. 第三阶段：闭环测试与风控（第 7 天）

| 序号 | 任务 | 产出 | 验收 |
|------|------|------|------|
| 3.1 | 频率限制落地 | 各平台动作前检查「每小时次数」 | 小红书 ≤10/时，抖音/京东 ≤30/时，超限则等待或跳过 |
| 3.2 | 文案随机化策略复核 | 文档或配置说明 | 确保无完全一致群发，必要时加随机句/emoji |
| 3.3 | 异常与人工干预 | 无法解析的表情/语音记录 + 提醒 | 写入 DB 或单独队列，并触发「需人工」通知 |
| 3.4 | 压力与稳定性测试 | 测试报告或清单 | 随机等待、错峰、24 小时试跑不封号（在测试账号上验证） |
| 3.5 | 邮件提醒联调 | 端到端测试 | status→成交 一次，确认邮件/Webhook 收到且内容正确 |

## 4. 风险与依赖（与 product 对齐）

- **频率限制**：必须在代码中写死或配置化，不可仅文档约定。
- **文案随机化**：所有外发文案必须经 AI 或随机模板生成，禁止硬编码同一条群发。
- **人工干预**：建议预留「人工接管」开关或队列，便于后续扩展。
- **依赖**：各平台后台若改版，需维护选择器/接口；API 额度与成本需单独规划。
